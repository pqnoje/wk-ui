Sobre

Este site é um teste da WK Techonoly para avaliar a codiffikcação de back-end, front-end, banco de dados e aspectos gerais.

Back-end

O back-end foi programado utilizando o framework Spring Boot 3.0.0.

Utililei JPA para fazer o cadastro de todos os candidados fornecidos pelo JSON que fica em './complete/dados.json'. Foi feito o cadastro assim que a aplicação termina Spring Boot termina.

Segue a lista de end-points do back-end:

GET http://localhost:8080/patients/by-state

Traz todos os candidatos separados por estado.

GET http://localhost:8080/patients/bmi-by-age-range

Traz os candidatos mostrando a média de IMC por intervalo de idade.

GET http://localhost:8080/patients/fat-by-gender

Traz a quantidade de obesos por gênero.

GET http://localhost:8080/patients/median-age-for-blood-type

Traz a média de idade por tipo sanguíneo.

GET http://localhost:8080/patients/blood-donors

Traz a quantidade de doadores por tipo sanguíneo.

Front-end

O front-end foi programado utilizando o framwork Angular 16.2.0, Swimlane(ngx-charts) 20.4.1 e ngx-bootstrap 11.0.2.

Utilizei funções do Angular HttpClient para fazer requisições no localhost da aplicação Java que fornece informações completas sobre os candidatos cadastrados na aplicação back-end.

O front-end utiliza bibliotecas Swimlane para fazer renderização dos graficos de barra, setor e cards. As informações foram processadas em cada componente para que atenda ao modelo de dados que o Swimlane necessita para construir as informações. O Swimlane foi escolhido pela forma que os tem de assinatura de dados que precisa passar como parâmetro.

Foi utilizado Angular Material com bootstrap para fazer a apresentação de alguns ícones e a construção do menu lateral.

Não há responsividade para os gráficos, eles são construídos para serem apresentados em desktop ou laptop. Pensei no layout do site(menu e topo) para serem responsivos. Para ler os gráficos pelo seu celular, é necessário navegar para a direita para vizualizar o gráfico de forma completa.

Banco de Dados

O banco de dados está em memória com o H2.

Não senti necessidade em pôr um banco de dados em um gerenciador instalado, como se trata de uma avaliação e a aplicação será encerrada após a avaliação e os dados não são alterados(apenas consultados), foi escolhido o banco de dados em memória apenas para armazenar os dados do arquivo JSON e serem consutados em tempo de execução da aplicação.

Aspectos Gerais

O código foi arquitetado para que respeite o CLEAN code e os princípios SOLID. A aplicação é bem pequena por isso não da para perceber os princípios SOLID. Isso é mais vizivel na camada front-end onde está a maior parte da lógica do sistema onde os componentes são responsáveis por trazer os dados dos candidatos, filtrar por caso de uso(tipo sanguíneo, idade, gênero e obesidade), e definidos para serem passados como parâmetro para os gráficos. Basicamente os componentes que têm a responsabilidade de construír a lógica de dados são a parte essencial para vizualizar a qualidade de código pois o back-end tem pouco trabalho. No back-end consegui pôr pouco código, mas é o suficiente para deixar o front-end responsável por abstrair os dados para que os gráficos sejam renderizados. Outro ponto importante, o estilo de código utilizado foi da maneira que acho necessária: sem ponto e vírgula, com funções de arrow function, com utilizaçõa de ECMAScript 6 e definição de tipo para a maioria dos modelos.

Não foram utilizados recursos avançados para programação do sistema.
